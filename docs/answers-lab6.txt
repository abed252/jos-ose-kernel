1️⃣ Q1: How does e1000_transmit() ensure safe descriptor usage?

The e1000_transmit() function first reads the current tail index of the transmit ring (E1000_TDT) and accesses the corresponding descriptor in tx_ring.  
It then checks if the descriptor's status has the E1000_TXD_STAT_DD bit set — which means the hardware has finished using this descriptor and it’s free.

This approach ensures:
- No overwrite happens while a packet is in-flight.
- The ring wraps around safely.
- It gracefully handles full-queue cases.

---

2️⃣ Q2: How does e1000_recv() ensure safe packet delivery?

In e1000_recv(), the function uses a static tail index to track where the next packet should be read from in rx_ring.  
It checks whether the descriptor at the current tail:
- Has the E1000_RXD_STAT_DD (descriptor done) bit set.
- Has the EOP (End of Packet) bit set (for simplicity, jumbo frames not supported).
- And that the received length fits within the user buffer.

If DD is not set, the queue is empty, so it returns -E_AGAIN to signal the user-level code to wait and retry later.

This approach ensures:
- Safe packet delivery.
- Correct reuse of receive descriptors.
- Graceful handling when no packet is ready.

🔧 How to Run and Test

1. Build the system:
   make

2. Launch Docker containers:
   docker-compose up --build

3. Start services:
   - Chat server:
     ./chatd
   - Chat clients:
     ./chatcli
   - File server:
     ./filed
   - File clients:
     ./filecli PUT <filename>
     ./filecli GET <filename>
   - Transferfile server:
     ./transferfile

4. Test from host (for transferfile server):
   Use:
   curl -X PUT --data-binary @localfile.txt http://<jos-ip>:<port>/filename.txt

5. Verify:
   - Check server logs.
   - Check file contents.
   - Check chat interactions.
   - Test edge conditions with multiple clients.
